1. What Happened: The Error Journey

After updating the app icon and splash screen in app.json, the build initially worked with expo prebuild --clean. But subsequent runs of pnpm run android failed with confusing errors that kept changing:

Error 1: Invalid <color> for given resource value in appcompat-1.7.1
Error 2: Invalid <color> for given resource value in material-1.13.0
Error 3: Directory does not exist: .../expo.modules.ui-56.0.13-runtime_dex
Success: Build finally succeeded after clearing build cache

Each cleanup attempt seemed to "fix" one error, only to reveal another lurking in the corrupted cache.

2. Core Concepts: Gradle, Maven, Transforms, and More

Gradle

The build tool used by Android projects. It's like a project manager that:

  • Downloads dependencies from repositories (via Maven or other formats)
  • Compiles code (Kotlin, Java, C++)
  • Processes resources (images, XML layouts, strings, colors)
  • Generates APKs/AABs (the final app package)

Maven

A dependency management format and protocol. Most Android libraries are published as Maven artifacts:

  • POM files (Project Object Model) — metadata describing dependencies
  • JAR files — compiled Java/Kotlin code
  • AAR files (Android Archive) — JAR + Android resources + native libraries

Gradle Transforms

When Gradle downloads a Maven artifact (like an AAR), it may need to transform it before use:

  • Resource merging — merging an AAR's resources with the app's resources
  • Desugaring — converting newer Java features to older Java versions
  • Dexing — converting compiled bytecode to Android DEX format

These transforms are cached at ~/.gradle/caches/[version]/transforms/[hash]/transformed/

AAPT2 (Android Asset Packaging Tool 2)

The tool that processes Android resources (XML layouts, drawables, colors, strings). It:

  • Parses resource XML files
  • Compiles them into binary format
  • Merges resources from all libraries and the app
  • Validates resource values (must be correct colors, dimensions, etc.)

Key behavior: AAPT2 is strict about color values — they must be hex (#RRGGBB or #AARRGGBB), not named colors like "black".

D8/Dexing

The tool that converts compiled Java/Kotlin bytecode (.class files) into Android DEX format (.dex files), which is what the Android runtime executes. Dexing is performed on:

  • App code — compiled from your source
  • Library code — extracted from AAR JARs

Gradle Build Cache

A separate cache from the transforms cache that stores build outputs (not source artifacts). When you run a build with --build-cache, Gradle can reuse:

  • Compiled code (if sources unchanged)
  • Merged resources (if resources unchanged)
  • Dexed bytecode (if bytecode unchanged)

This cache is at ~/.gradle/caches/build-cache-*

3. Root Cause Analysis: Docker Corruption

The root cause of all these errors was Docker experiments you ran earlier:

You created Docker containers to build the Android app. Inside the container, Gradle ran as root (Linux default for Docker). When Gradle cached artifacts in /home/abhi/.gradle/caches/, those files became owned by root, not by your user.

Why This Caused Corruption

  1. Permission conflicts: When you later ran Gradle as your user (abhi), it couldn't write to root-owned directories
  2. Partial writes: Some transforms may have started but couldn't complete due to permission errors
  3. Stale metadata: Gradle's metadata (results.bin) recorded successful transforms, but the actual output files were missing or corrupted
  4. Hash collisions: Gradle uses content hashes to identify transforms. Corrupted hashes made Gradle think artifacts were valid when they weren't

The Cascade of Failures

Docker root-owned files
  ↓
Gradle can't write as abhi
  ↓
Transforms partially complete
  ↓
Metadata says "done", files are missing
  ↓
AAPT2 tries to use corrupted resources → Error #1
  ↓
Clean transform, Gradle re-downloads
  ↓
Re-download also corrupted → Error #2
  ↓
Dex merger tries to use corrupted dexes → Error #3
  ↓
Finally clear build cache → Success!

5. What We Tried (And Why It Failed)

Attempt 1: ./gradlew clean

  • What it does: Deletes project build directory
  • Why it failed: Doesn't touch Gradle's user cache (~/.gradle/caches/)

Attempt 2: rm -rf ~/.gradle/caches/9.3.1/transforms

  • What it does: Deletes transform cache
  • Why it failed: Gradle re-downloads artifacts, but the build cache still has stale references to old transforms

Attempt 3: pkexec rm -rf ~/.gradle/caches/9.3.1

  • What it does: Tries to delete entire Gradle cache as root
  • Why it failed: pkexec requires authentication (password). Without a TTY, it fails silently

Attempt 4: ./gradlew --stop + delete transforms

  • What it does: Stops Gradle daemon (in-memory cache) + deletes transforms
  • Why it failed: Build cache still had stale dex file references

Attempt 5: --no-build-cache

  • What it does: Skips using build cache for this build
  • Why it failed: The project-level cache was skipped, but the global cache still had corrupted metadata

Attempt 6: rm ~/.gradle/caches/build-cache-* + --no-build-cache + daemon stop

  • What it does: Deletes global build cache + disables caching for this run + clears daemon
  • Why it succeeded: All stale metadata cleared; transforms re-run from scratch; no cached dex references

6. The Fix: Two-Part Solution

The final fix required both parts:

Part 1: Fix the Splash Screen Color

// BEFORE (app.json)
"backgroundColor": "black"  // ❌ AAPT2 doesn't accept named colors

// AFTER (app.json)
"backgroundColor": "#000000"  // ✅ Valid hex color

Why: The expo-splash-screen plugin generates a color resource from the app.json value. AAPT2 requires hex values, not named colors like "black". This fixed the "Invalid <color>" error.

Part 2: Clear Corrupted Build Cache

rm -rf ~/.gradle/caches/build-cache-*
cd android
./gradlew --stop
./gradlew app:assembleDebug --no-build-cache

Why: The build cache had stale references to corrupted transform outputs. Clearing it forced Gradle to re-run all transforms and dex merges from scratch.

7. Lessons Learned

  1. Never run Android builds in Docker as root: Always specify a user with proper permissions to avoid owning cached files as root
  2. Clean caches hierarchically: When troubleshooting builds, clear in this order:
    • Project build directory (./android/app/build)
    • Project Gradle cache (./android/.gradle)
    • Global transforms cache (~/.gradle/caches/[version]/transforms)
    • Build cache (~/.gradle/caches/build-cache-*)
    • Daemon (./gradlew --stop)
  3. AAPT2 is strict: Always use hex colors in Android resources, even in JSON configs that generate resources
  4. pkexec without TTY fails silently: Use sudo with password or configure pkexec to allow specific commands
  5. Error cascades in builds: One corrupted file can cause multiple different errors as the build progresses through different stages
